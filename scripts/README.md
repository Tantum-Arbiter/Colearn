# CMS (Content Management System)

> **For LLMs / AI agents**: This README is the authoritative reference for the story CMS pipeline.
> Read this file before modifying any CMS scripts, upload logic, or Firestore schema.
> If you change the pipeline, **update this file**.

## Overview

The CMS is a file-based content pipeline that turns story JSON files + image assets into live content
available to mobile clients. There is **no admin UI** — stories are authored as files in this repo,
validated by CI, and deployed automatically on merge to `main`.

### Pipeline At a Glance

```
Author story-data.json + images in scripts/cms-stories/{story-id}/
         ↓
cms-manager CLI validates & formats locally
         ↓
git push → PR → merge to main
         ↓
GitHub Actions (cms-stories-sync.yml)
  ├─ Job 1: Validate JSON against story-schema.json (always runs)
  ├─ Job 2: gsutil rsync images → GCS bucket (delta, checksums)
  └─ Job 3: upload-stories-to-firestore.js → Firestore (delta, checksums)
             upload-assets-to-firestore.js → asset version metadata
         ↓
Mobile app launch
  ├─ VersionManager.checkVersions() — integer version comparison
  ├─ BatchSyncService.performBatchSync() — POST /api/stories/delta
  │     (sends client story checksums, receives only changed stories)
  ├─ CacheManager downloads new images via signed URLs from GCS
  └─ StoryLoader merges bundled + CMS stories for rendering
```

## Scripts

### upload-stories-to-firestore.js

Reads all `story-data.json` files, calculates SHA-256 checksums, and uploads changed stories
to the Firestore `stories` collection using **delta-sync** (only changed stories are written).

**Key design decisions:**

1. **Firestore batch chunking** — Firestore has a hard 500-operation limit per batch. Stories are
   chunked into batches of **400** to stay safely under the limit. At 1000 stories this means 3
   sequential commits instead of a crash. The same chunking applies to orphan deletion.

2. **Delta-sync via checksums** — Each story gets a SHA-256 checksum computed from its content
   (id, title, localized fields, page text, images, interactive elements, music challenges).
   The script compares against checksums stored in `content_versions/current`. Only stories with
   changed checksums are written, saving Firestore write costs.

3. **Orphan deletion** — Stories present in Firestore but not in source files are deleted.
   The `content_versions/current` document is replaced entirely (not merged) so deleted story
   checksums are automatically removed.

4. **Upload modes** — `UPLOAD_MODE` controls which stories are read:
   - `cms-only` (default): Only `scripts/cms-stories/`
   - `bundled`: Only `grow-with-freya/assets/stories/`
   - `all`: Both directories

**Environment variables:**

| Variable | Required | Description |
|----------|----------|-------------|
| `GCP_SA_KEY` | Yes (CI) | Base64-encoded GCP service account JSON |
| `FIREBASE_SERVICE_ACCOUNT_KEY_PATH` | Yes (local) | Path to service account JSON file |
| `FIREBASE_PROJECT_ID` | No | Defaults to `apt-icon-472307-b7` |
| `UPLOAD_MODE` | No | `cms-only` (default), `bundled`, or `all` |
| `DRY_RUN` | No | `true` to validate without writing |
| `FORCE_UPLOAD` | No | `true` to ignore checksums and re-upload everything |
| `FIRESTORE_EMULATOR_HOST` | No | e.g. `localhost:8082` for local dev |

### upload-assets-to-firestore.js

Reads MD5 checksums from **GCS object metadata** for all assets in the bucket and writes them
to `asset_versions/current` in Firestore. This is used by the client for asset version tracking.

**Key design decision: GCS metadata instead of file downloads** — GCS automatically computes and
stores an MD5 hash for every uploaded object. This script reads `file.metadata.md5Hash` instead of
downloading files to compute checksums locally. At 1,700+ assets this avoids downloading ~340MB
during every CI run. The checksums are converted from base64 to hex for consistency.

**Note:** The client never reads individual asset checksums. It only compares the integer asset
version number. A version bump triggers a delta-sync which compares story-level checksums. If no
stories changed, zero assets are downloaded — even if the asset version bumped.

## Firestore Schema

### `stories` Collection

One document per story. Document ID = story ID (kebab-case).

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique identifier, e.g. `squirrels-snowman` |
| `title` | string | English display title |
| `localizedTitle` | map | `{en: "...", pl: "...", es: "...", ...}` (14 languages) |
| `category` | string | One of: personalized, bedtime, adventure, nature, friendship, learning, fantasy, music, activities, growing |
| `tags` | string[] | Filter tags (calming, bedtime, adventure, learning, music, etc.) |
| `description` | string | English description |
| `localizedDescription` | map | Localized descriptions |
| `coverImage` | string | GCS asset path for cover |
| `ageRange` | string | e.g. `"2-5"` |
| `duration` | integer | Number of pages |
| `isPremium` | boolean | Premium content flag |
| `version` | integer | Story content version |
| `checksum` | string | SHA-256 of story content for delta-sync |
| `pages` | array | Page objects (see below) |
| `createdAt` | timestamp | Auto-set on upload |
| `updatedAt` | timestamp | Auto-set on upload |

### Page Object

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique page ID (kebab-case) |
| `pageNumber` | integer | 0-based sequential |
| `type` | string | `cover` or `story` |
| `text` | string | English page text |
| `localizedText` | map | `{en: "...", pl: "...", ...}` |
| `backgroundImage` | string | GCS asset path |
| `interactionType` | string | `none`, `interactive_state_change`, or `music_challenge` |
| `interactiveElements` | object | Before/after state config (tap-to-reveal pages) |
| `musicChallenge` | object | Music challenge config (see MUSIC_FEATURE.md) |

### `content_versions` Collection

Singleton document `current`. Replaced entirely on each upload.

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Always `"current"` |
| `version` | integer | Increments on every change |
| `totalStories` | integer | Count of active stories |
| `storyChecksums` | map | `{storyId: "sha256hash", ...}` |
| `lastUpdated` | timestamp | Last upload time |

### `asset_versions` Collection

Singleton document `current`. Tracks GCS asset checksums.

| Field | Type | Description |
|-------|------|-------------|
| `version` | integer | Increments on asset changes |
| `totalAssets` | integer | Count of files in GCS |
| `totalSizeBytes` | integer | Total size of all assets |
| `assetChecksums` | map | `{gcsPath: "md5hex", ...}` |
| `lastUpdated` | timestamp | Last verification time |

## Story Schema (story-schema.json)

All `story-data.json` files are validated against `scripts/story-schema.json` in CI.

**Required fields:** `id`, `title`, `category`, `pages` (with at least 1 page).
**Per page required:** `id`, `pageNumber`, `text`.

**Validation rules:**
- `id` must be kebab-case (`^[a-z0-9-]+$`)
- `title` must be 1–100 characters
- `category` must be one of the 10 defined categories
- `tags` must be from the 15 defined filter tags
- Page numbers must be sequential starting from 0
- Music challenge pages must include all required fields

## Localization

All stories support **14 languages**: en, pl, es, de, fr, it, pt, ja, ar, tr, nl, da, la, zh.

The `LocalizedText` interface is used for titles, descriptions, and per-page text:

```json
{
  "localizedTitle": {
    "en": "The Squirrel's Snowman",
    "pl": "Bałwan Wiewiórki",
    "es": "El Muñeco de Nieve de la Ardilla"
  }
}
```

English (`en`) is always required and used as the fallback. Other languages are optional.
The app calls `getLocalizedText()` from `types/story.ts` to resolve the correct language.

## GCS Bucket

**Bucket:** `gs://colearnwithfreya-assets/stories/`

Assets are synced via `gsutil -m rsync -r -c` which:
- `-m` = parallel uploads for speed
- `-r` = recursive (all subdirectories)
- `-c` = compare checksums (not timestamps) for accurate delta
- `-d` flag is used to delete files from GCS that no longer exist in source

Clients never access GCS directly. The gateway generates **1-hour signed URLs** via
`POST /api/assets/batch-urls` which the client uses to download images.

## Local Development

```bash
cd scripts
npm install

# Upload to real Firestore (requires service account key)
FIREBASE_SERVICE_ACCOUNT_KEY_PATH=./service-account-key.json \
FIREBASE_PROJECT_ID=apt-icon-472307-b7 \
node upload-stories-to-firestore.js

# Dry run (validate only, no writes)
DRY_RUN=true node upload-stories-to-firestore.js

# Force re-upload all stories
FORCE_UPLOAD=true node upload-stories-to-firestore.js

# Upload to Firestore emulator
FIRESTORE_EMULATOR_HOST=localhost:8082 node upload-stories-to-firestore.js
```

